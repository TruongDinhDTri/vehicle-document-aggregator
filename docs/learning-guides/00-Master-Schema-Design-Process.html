"We're building a food delivery app. People can browse restaurants, look at their menus, and place orders for delivery. Design the database schema."

🧱 The 5 Schema Patterns — Lego Blocks of Any Backend

🎯

Mental Model: Most real systems are just combinations of these 5 patterns. Recognize them when reading requirements → schema design becomes almost automatic. Think of them as Lego pieces — you assemble, not invent.

1️⃣ Core Entity Pattern — The Main Objects

Idea: Represents the main objects in the system. Usually nouns from the requirements.

Mental Rule: When reading requirements, look for nouns.

Requirement: "Users can create posts"
Entities:    User, Post

Requirement: "Users buy products"
Entities:    User, Product, Review

Common Core Entities:
  User    Product    Post    Order
  Comment Course     File    Project

Schema Shape:

User
─────
id (PK)
name
email
created_at

Post
─────
id (PK)
user_id (FK)   ← ownership
title
content
created_at
💡

Interview move: "First, let me identify the core entities from the requirements — the main nouns: User, Product, Order..."

2️⃣ One-to-Many Pattern — Parent → Children

Idea: One entity can have many related records. Most common pattern in databases.

Mental Rule: "A [parent] can have multiple [children]" → FK lives on the MANY side.

User → many Orders
Post → many Comments
Category → many Products
Restaurant → many MenuItems

FK always lives on the "many" side:

Order                Comment
──────               ──────
id (PK)              id (PK)
user_id (FK)  ←      post_id (FK)  ←
created_at           content
total_price          created_at

"The child holds the reference to the parent"
💡

Quick rule: The "many" side gets the ForeignKey. Order belongs to User → Order.user_id, not User.order_id. Imagine storing an array in User — that violates 1NF.

3️⃣ Many-to-Many Pattern — Join Table

Idea: Both sides can have multiple relationships. Cannot store this directly — must create a junction table.

Mental Rule: "many ↔ many" → create a join table.

Students take many Courses AND Courses have many Students
Users like many Posts AND Posts are liked by many Users
Products have many Tags AND Tags apply to many Products

Simple join (no extra data):

Enrollment (junction)
──────────────────────
student_id (FK)
course_id  (FK)
enrolled_at           ← if you need this, it becomes a through model!

Through model (relationship has its own data):

OrderItem (through model — Order ↔ Product)
───────────────────────────────────────────
order_id   (FK)
product_id (FK)
quantity              ← belongs to the relationship
unit_price            ← belongs to the relationship
💡

Signal for through model: If relationship data gets "???" when placed in either table alone → it belongs to the junction. See Through Models section →

4️⃣ Activity / Event Pattern — History Tables

Idea: Represents actions or events that happen over time. These tables store system history and are usually append-only.

Mental Rule: If something happens in the system → it's an event table. Look for verbs.

User logs in         → LoginEvent
User places order    → Order
User sends message   → Message
User uploads file    → Upload
Payment processed    → Payment
Notification sent    → Notification

Schema shape — always has timestamp:

Order (Activity)
────────────────
id (PK)
user_id (FK)
total_price
status           ← state transitions
created_at       ← ALWAYS timestamp
updated_at
💡

Characteristics: Append-only (rarely UPDATE or DELETE). Usually has created_at, status. Often indexed on created_at for time-range queries. Perfect for audit logs, notifications, order history.

5️⃣ Ownership Pattern — Who Created/Owns What

Idea: Represents who owns or created something. Usually implemented as a ForeignKey (a specialized One-to-Many).

Mental Rule: "User owns X" → FK on X pointing back to User.

User owns Posts      → Post.user_id (FK to User)
User owns Orders     → Order.user_id (FK to User)
User owns Files      → File.uploaded_by (FK to User)
User owns Projects   → Project.owner_id (FK to User)

Creates the pattern:

User 1:N Resource

"owner_id", "created_by", "uploaded_by" — all same pattern,
different semantic naming for clarity
💡

Note: Ownership is technically One-to-Many. It's called out separately because it's so universal — almost every table in a multi-user system has an owner column. Naming it clearly ("owner_id" vs "user_id") signals domain intent.

🔗 How Systems Are Built — Composition Examples

Most real apps are just these 5 patterns combined. Once you see this, system design feels much less intimidating.

SystemEntitiesPatterns Used
Social Media User, Post, Comment, Like, Follow Ownership (User→Post), One-to-Many (Post→Comments), Many-to-Many (Users Like Posts, Users Follow Users), Activity (Comments, Likes)
E-commerce User, Product, Order, OrderItem, Review, Category Ownership (User→Order), One-to-Many (Order→OrderItem, Product→Reviews), Many-to-Many through OrderItem, Activity (Orders, Reviews)
LMS (Courses) User, Course, Lesson, Enrollment, Progress Core Entities, One-to-Many (Course→Lessons), Many-to-Many through Enrollment (grade + date), Activity (Progress)
Food Delivery User, Restaurant, MenuItem, Order, OrderItem, Driver Ownership, One-to-Many (Restaurant→Menu), Many-to-Many through OrderItem, Activity (Order status changes)
🎯

Interview Move — Say This: "When I see a system design problem, I first identify which of the 5 patterns apply: I can see Core Entities [X, Y, Z], they connect via One-to-Many [A→B] and Many-to-Many [C↔D], and there are Activity tables for [E, F]. Let me sketch the schema..."

This shows pattern recognition, not memorization. Interviewers love it. 🔥

2. Step 0: Clarify Requirements

💡

Interview Tip: The #1 mistake is jumping straight to drawing tables. Interviewers WANT you to ask questions first. This shows you think before you code.

Organize your clarifying questions into 6 buckets. This framework works for ANY schema design problem:

The 6-Bucket Framework

BucketWhat to AskFood Delivery Example
1. Users & Roles Who uses this system? What types of users? Customer, Restaurant Owner (no delivery driver for MVP)
2. Core Actions What can each user type DO? Customer: browse, order. Owner: manage menu
3. Constraints Rules/limits? How many X per Y? 1 order = 1 restaurant. ~50 menu items per restaurant
4. Business Rules What must NEVER break? Old orders keep old prices even if restaurant changes price
5. Edge Cases What happens when things change/fail? Deleted menu item? Old orders still show the item name
6. Scale How many users? What DB technology? 10K users, 500 restaurants. PostgreSQL is fine, no sharding needed
Example Clarifying Questions to Ask
  1. "Who are the user types? Just customers, or also restaurant owners and delivery drivers?"
  2. "Can a customer order from multiple restaurants in one order?"
  3. "What happens to an order if a restaurant changes their price or deletes a menu item?"
  4. "Do we need to track order status? What are the statuses?"
  5. "How many restaurants and users are we expecting? Do we need to think about sharding?"
  6. "Does a restaurant owner manage their own menu, or is there a separate admin role?"
Food Delivery: Clarified Requirements

Users & Roles

  • Customer and Restaurant Owner

Core Actions

  • Customer can browse restaurants, view menus, and place orders
  • Restaurant Owner can add, update, and remove menu items

Constraints

  • One Order = One Restaurant only
  • A customer can place unlimited orders over time
  • Order statuses: PENDING, READY, DELIVERING, DELIVERED, CANCELLED
  • ~50 menu items per restaurant

Critical Business Rules

  • If a restaurant changes their price, old orders must still show the old price

Edge Cases

  • Orders can be cancelled
  • If a restaurant deletes a menu item, old orders must still show the item name

Scale

  • ~10,000 users, ~500 restaurants
  • PostgreSQL. No need for sharding or NoSQL
⚠️

Common Mistake: Skipping this step and jumping to "User table, Order table..." The business rules about price snapshots and deleted menu items are critical — they change your entire schema design. You'd never discover them without asking.

3. Step 1: Access Patterns

🔎

Key Idea: Think screen-by-screen. Imagine the user opening the app — what do they see? What data does each screen need?

Access patterns tell you how the data will be read and written. This directly influences your indexes, relationships, and whether you need denormalization.

Screen-by-Screen Thinking

#Screen / ActionAccess PatternType
AP1Customer opens appGet all restaurantsREAD
AP2Tap on a restaurantGet restaurant's menu itemsREAD
AP3Tap on a menu itemGet menu item detailREAD
AP4Owner edits menuUpdate/create/delete menu itemWRITE
AP5Customer places orderCreate order + order items with price snapshotsWRITE
AP6Customer views orderGet order details with all itemsREAD
AP7Owner: incoming ordersGet all orders for restaurant, filtered by statusREAD
📚

Why This Matters:

  • AP5 tells you that you need price snapshots (copy price at time of order)
  • AP7 tells you that status needs a db_index because it's filtered frequently
  • AP2 tells you MenuItem needs a FK to Restaurant so you can query by restaurant
⚠️

Common Mistake: Only thinking about write patterns. The majority of database operations are reads. If your access pattern says "filter orders by status for a restaurant", that tells you exactly which fields need indexes.

4. Step 2: Identify Entities

🎯

The Technique: "Look for nouns" in the requirements. Each noun is a candidate entity. Then filter using the entity test.

The "Look for Nouns" Technique

Read through your requirements and highlight every noun:

"People can browse restaurants, look at their menus, and place orders for delivery."

Candidates: People (Customer), Restaurant, Menu (MenuItem), Order, Delivery

Entity vs. Field — The Test

✅

Something is an entity if:

  1. It has its own attributes (multiple fields)
  2. It gets referenced by other things (other entities point to it)
  3. You need to query it independently

If it fails all three — it's just a field on another entity.

CandidateHas Own Attributes?Gets Referenced?Query Independently?Verdict
Customername, email, phoneBy OrderYesEntity
Restaurantname, addressBy MenuItem, OrderYesEntity
MenuItemname, priceBy OrderItemYes (per restaurant)Entity
Orderstatus, total, timestampsBy OrderItemYesEntity
OrderItemquantity, price_snapshotBy OrderYes (line items)Entity
DeliveryJust a status on Order?NoNo (for MVP)Field
"Menu"No (it's a list of items)NoNoNot an entity
💡

Interview Tip: "Menu" is NOT an entity — it's just the collection of MenuItems for a restaurant. A common mistake is creating a Menu table that just holds a restaurant FK. The MenuItems themselves already have the restaurant FK, making a separate Menu table redundant.

5. Step 3: Identify Relationships

🔗

The Golden Rule: "FK goes on the many side." Always. No exceptions.

The Receipt Analogy

Think of a receipt at a restaurant:

  • One customer can have many receipts (orders)
  • Each receipt says "Customer: John" on it — the receipt (the many side) holds the reference to the customer
  • You don't go to John's profile and list every receipt number — that would be a comma-separated mess

So: Order.customer_id (FK on Order, the many side) → NOT Customer.order_ids

Relationship Types

TypeRuleFK LocationExample
1:1 One X has exactly one Y FK on either side (+ unique=True) User ↔ Profile
1:N One X has many Y FK on the "many" side (Y) Restaurant → many MenuItems
M:N Many X have many Y Separate "through" / junction table Order ↔ MenuItem (via OrderItem)

Food Delivery Relationships

Customer   ──< Order        # 1:N  (one customer, many orders)
Restaurant ──< Order        # 1:N  (one restaurant, many orders)
Restaurant ──< MenuItem     # 1:N  (one restaurant, many menu items)
Order      ──< OrderItem    # 1:N  (one order, many line items)
MenuItem   ──< OrderItem    # 1:N  (one menu item, in many order items)
💡

OrderItem is a "through table" — it connects Order and MenuItem in a M:N relationship, but with extra data (quantity, price snapshot). In Django, this is a model with two ForeignKeys plus additional fields. You can't use a plain ManyToManyField when you need extra data on the relationship.

⚠️

Common Mistake: Putting the FK on the wrong side. If you write Customer.order_id, that means a customer can only have ONE order ever. Always ask: "Which side is many?" and put the FK there.

6. Step 4: Define Attributes

For each entity, define: PK (auto), FKs (from relationships), and descriptive fields.

The Snapshot Pattern

📸

Critical Concept: Some data must be frozen at the time of action. If a restaurant changes their "Pho" from $12 to $15, your old order must still show $12. This means copying (snapshotting) certain values into the order.

What to Snapshot

FieldWhy Snapshot?Where
priceRestaurant can change prices anytimeOrderItem.price
item_nameRestaurant can rename or delete itemsOrderItem.item_name

This means OrderItem has both a FK to MenuItem (for traceability) AND its own price and item_name fields (for historical accuracy).

Complete Attribute List

# Entity: Attributes

Customer:    id(PK), name, email, phone
Restaurant:  id(PK), name, address
MenuItem:    id(PK), restaurant_id(FK), name, price
Order:       id(PK), customer_id(FK), restaurant_id(FK),
             status, total_price, created_at, updated_at
OrderItem:   id(PK), order_id(FK), menu_item_id(FK),
             quantity, price, item_name   # price & item_name = snapshots!
⚠️

Common Mistake: Not snapshotting. If OrderItem only has menu_item_id without its own price and item_name, then when the restaurant changes the price, ALL historical orders change retroactively. This is a data integrity disaster.

8. Step 6: Convert to Django Models

💻

Key Decisions: on_delete choice, db_index placement, choices for enums, and __str__ for debugging.

The Complete Food Delivery Schema

from django.db import models


class Customer(models.Model):
    name = models.CharField(max_length=100)
    email = models.EmailField(unique=True)
    phone = models.CharField(max_length=15)

    def __str__(self):
        return self.name


class Restaurant(models.Model):
    name = models.CharField(max_length=100)
    address = models.CharField(max_length=255)

    def __str__(self):
        return self.name


class MenuItem(models.Model):
    restaurant = models.ForeignKey(Restaurant, on_delete=models.CASCADE)
    name = models.CharField(max_length=100)
    price = models.DecimalField(max_digits=10, decimal_places=2)

    def __str__(self):
        return self.name


class Order(models.Model):
    customer = models.ForeignKey(Customer, on_delete=models.CASCADE)
    restaurant = models.ForeignKey(Restaurant, on_delete=models.CASCADE)
    status = models.CharField(max_length=20, db_index=True, choices=[
        ('PENDING', 'Pending'),
        ('READY', 'Ready'),
        ('DELIVERING', 'Delivering'),
        ('DELIVERED', 'Delivered'),
        ('CANCELLED', 'Cancelled'),
    ])
    total_price = models.DecimalField(max_digits=10, decimal_places=2, default=0)
    created_at = models.DateTimeField(auto_now_add=True, db_index=True)
    updated_at = models.DateTimeField(auto_now=True)

    def __str__(self):
        return f"Order #{self.id}"


class OrderItem(models.Model):
    order = models.ForeignKey(Order, on_delete=models.CASCADE)
    menu_item = models.ForeignKey(MenuItem, on_delete=models.SET_NULL, null=True)
    quantity = models.IntegerField(default=1)
    price = models.DecimalField(max_digits=10, decimal_places=2)       # snapshot!
    item_name = models.CharField(max_length=100)                        # snapshot!

    def __str__(self):
        return self.item_name

on_delete Choices — When to Use Each

ChoiceWhat HappensWhen to UseExample
CASCADE Delete parent → delete all children Children make no sense without parent Delete Order → delete its OrderItems
SET_NULL Delete parent → set FK to NULL Child has its own value (snapshots) and should survive Delete MenuItem → OrderItem.menu_item = NULL (but item_name & price snapshots remain)
PROTECT Prevent deletion if children exist Deletion would cause data loss or business rule violation Can't delete a Customer who has orders
SET_DEFAULT Set FK to a default value Rare — when you have a fallback entity Deleted user → assign orders to "Anonymous" user
✅

Key Decision in Our Schema:

  • OrderItem.menu_item uses SET_NULL because the OrderItem has snapshots (price, item_name) and should survive even if the original menu item is deleted
  • OrderItem.order uses CASCADE because an order item makes no sense without its order
  • MenuItem.restaurant uses CASCADE because if a restaurant is deleted, its menu items should go too

Other Django Model Patterns

db_index — When to Add It

Add db_index=True to fields you filter, sort, or lookup frequently:

  • status — filtered in AP7 (incoming orders by status)
  • created_at — sorted for "recent orders"
  • email — looked up for login (already indexed via unique=True)

Don't index everything. Each index slows down writes. Only index fields in your access patterns.

choices Field — Enumerated Values 
status = models.CharField(max_length=20, choices=[
    ('PENDING', 'Pending'),      # (stored_value, display_label)
    ('READY', 'Ready'),
    ('DELIVERING', 'Delivering'),
    ('DELIVERED', 'Delivered'),
    ('CANCELLED', 'Cancelled'),
])
  • First value = what's stored in the DB ('PENDING')
  • Second value = human-readable label for forms/admin ('Pending')
  • Validation happens at Django level, NOT database level (use constraints for DB-level)
__str__ — Why It Matters

Without __str__, Django admin and shell show: Order object (1). With it:

>>> Order.objects.first()
Order #1

>>> MenuItem.objects.first()
Pho Bo

Always define __str__ — it saves hours of debugging.

9. Step 7: Normalization Check

📐

Core Principle: "One table = one entity. Each field depends on the PK, the whole PK, and nothing but the PK."

1NF — Atomic Values

Rule: Every column stores a single, indivisible value. No arrays, no comma-separated lists.

Violates 1NF:

| order_id | items              |
|----------|--------------------|
| 1        | "Pho, Bun Bo, Che" |

Correct (1NF):

Order table + separate
OrderItem table, one row
per item.

2NF — Full Dependency on Whole PK

Rule: Every non-key field depends on the entire primary key. (Matters when PK is composite.)

❌

Violates 2NF: If OrderItem had composite PK (order_id + menu_item_id) and also stored restaurant_name — restaurant_name only depends on menu_item_id, not the full composite key.

Fix: Keep restaurant_name only in the Restaurant table.

3NF — No Transitive Dependencies

Rule: Non-key attributes should depend only on the primary key, not on other non-key fields.

🔑

Core Principle: Một bảng = một entity. Trong bảng chỉ nên có: (1) thuộc tính của entity đó, và (2) foreign key (pointer) để liên kết. Không bao giờ chứa thuộc tính của entity khác.

"Phụ thuộc" (Depends on) nghĩa là gì?

Đừng hiểu "phụ thuộc" là "nô lệ" hay "cần ai đó để sống."

Trong Database, nó chỉ có nghĩa:

"Đưa tôi thông tin A, tôi tra ra được CHÍNH XÁC thông tin B."
Đưa customer_id (C01)
  → tra ra customer_name
    ("Nguyễn Văn A")

# Tên "phụ thuộc" vào Mã KH.
# Biết ID → biết tên
Đưa order_id (#123)
  → tra ra order_date
    (07/03/2026)

# Ngày mua "phụ thuộc" vào
# Mã hoá đơn.

Đơn giản: biết A → biết B. Vậy B phụ thuộc vào A.

Tại sao customer_id lại phụ thuộc vào order_id?

Tưởng tượng bạn là thu ngân, cầm trên tay 1 tờ hoá đơn.

📋

Hoá đơn số #123 → gõ vào hệ thống → hệ thống nhả ra: "Khách hàng C01 mua đơn này."

Biết order_id → tra ra customer_id.
→ customer_id phụ thuộc vào order_id.

Đơn giản: tờ hoá đơn đó PHẢI thuộc về một khách hàng cụ thể.

Một bảng = Một entity — Chỉ có 2 loại cột hợp lệ 
Orders     → đại diện cho Order

Chỉ nói: "entity này liên quan tới entity nào"

Điều KHÔNG được phép (vi phạm 3NF)

Không được đặt THUỘC TÍNH CỦA ENTITY KHÁC trong bảng.

❌

SAI:

Orders
----------
order_id

customer_name   ← thuộc về Customer, không phải Order
customer_email  ← thuộc về Customer, không phải Order
Tại sao sai? — Ví dụ "Căn Nhà Cho Thuê"
🏠

Bảng Order giống như một CĂN NHÀ CHO THUÊ.

  • Chủ nhà (Khoá chính): order_id
  • Người thuê trọ: customer_id
  • Vợ của người thuê: customer_name

Bạn gõ cửa nhà #123. Cánh cửa mở ra, thấy cô vợ (customer_name) ngồi xem tivi.

VỀ HIỆN TƯỢNG:

Gõ cửa nhà #123 là thấy cô vợ. Cảm giác rất "trực tiếp."

VỀ BẢN CHẤT:

Cô vợ là vợ của anh chồng (customer_id), không phải vợ của căn nhà.

🚪

Nếu anh chồng dọn đi chỗ khác → cô vợ đi theo anh chồng, không ở lại với căn nhà #123.

→ customer_name KHÔNG QUAN TÂM tới order_id. Nó chỉ "đi ké" anh chồng vào ở trọ.

Chuỗi Bắc Cầu (Transitive Dependency)

Khi nhét customer_name vào bảng Order, mối quan hệ đi theo chuỗi:

order_id  →  customer_id  →  customer_name
   A      →      B        →       C

# 3NF ghét nhất là bọn "án theo bắc cầu."

customer_name nằm trong bảng Order, nhưng nó chẳng thèm quan tâm tới order_id.

Tên khách hàng là do customer_id quyết định, chứ mã hoá đơn số mấy làm sao quyết định được tên của người ta!

💡

Luật 3NF: Nếu một cột không chơi trực tiếp với chủ nhà (PK), mà lại phụ thuộc vào một đứa khác trong nhà → TÁCH NÓ RA Ở RIÊNG.

The Rule (English): If A → B → C, then C doesn't belong in A's table. Move C to B's table.

Cách Sửa — Đập ra làm 2 bảng
✅ 
Orders                      Customers
----------                  ----------
order_id                    customer_id
order_date                  customer_name
customer_id (FK)            customer_email
  • Order chỉ giữ customer_id (pointer) để biết hoá đơn này của ai.
  • Tên và email nằm bên bảng Customer, sửa 1 chỗ là xong.
❌

Ví dụ vi phạm với food delivery schema:

| order_id | restaurant_id | restaurant_name | status |
# restaurant_name phụ thuộc restaurant_id, không phải order_id
# Chuỗi: order_id → restaurant_id → restaurant_name

Sửa: restaurant_name chỉ nằm trong bảng Restaurant. Bảng Order chỉ giữ restaurant_id FK.

Interview Answer (English)

"Every field in a table should describe that table's entity, not another table's entity. If it describes another entity, it belongs in that other table — just reference it with a foreign key."

The Snapshot Exception

💡

Khớan, OrderItem.price có vi phạm 3NF không? Trông nó giống transitive dependency (order → menu_item → price). Nhưng đây là ngoại lệ có chủ đích, gọi là snapshot denormalization.

OrderItem.price KHÔNG phải "giá hiện tại của món ăn" — nó là "giá tại thời điểm đặt hàng." Nó là một sự thật khác tình cờ cùng tên. MenuItem.price (giá hiện tại) và OrderItem.price (giá lúc mua) có thể khác nhau. Đó là toàn bộ ý nghĩa.

For our food delivery app: PostgreSQL is the right choice. Orders have strong relationships with customers, restaurants, and items. ACID transactions ensure an order is either fully created or not at all. At 10K users / 500 restaurants, SQL handles this easily.

Indexes — The Phone Contacts Analogy

📞

Analogy: An index is like sorting your phone contacts by name. Without sorting, finding "John" means scrolling through ALL contacts one by one. With contacts sorted alphabetically, you can jump straight to "J" section.

A database index works the same way: instead of scanning every row (full table scan), the database can jump straight to matching rows.

How B-Tree Indexes Work (Technical Detail)

Without index:  Scan ALL rows  →  O(n) = slow
With B-Tree:    Binary search   →  O(log n) = fast

Example with 1,000,000 orders:
  Without index: check ~1,000,000 rows
  With index:    check ~20 comparisons (log2 of 1M ≈ 20)

What to Index in Our Schema

FieldWhyAccess Pattern
Order.statusFiltered frequentlyAP7: Get orders by status
Order.created_atSorted for recencyAP6: Recent orders first
Customer.emailLogin lookupAlready indexed via unique=True
All ForeignKeysJOINs and lookupsDjango auto-indexes FKs
⚠️

Don't over-index! Every index:

  • Speeds up reads (SELECT, WHERE, ORDER BY)
  • Slows down writes (INSERT, UPDATE, DELETE — index must be updated too)
  • Uses disk space

Only index fields that appear in your access patterns' WHERE/ORDER BY clauses.

Denormalization for Performance

Example: Order.total_price

Without Denormalization

# Must calculate every time:
SELECT SUM(price * quantity)
FROM order_items
WHERE order_id = 42;

# Slow if done on every page load

With Denormalization

# Pre-calculated, stored on Order:
total_price = models.DecimalField(
    max_digits=10,
    decimal_places=2,
    default=0
)

# Fast: just read one field
💡

The Trade-off: Denormalization trades write complexity for read speed. You must now update total_price every time an OrderItem is added/modified. But you avoid expensive SUM queries on every read. For read-heavy data (like order totals displayed on every page), this is worth it.

11. Step 9: Best Practices & Checklist

Soft Deletes

Instead of actually deleting records, mark them as deleted:

class MenuItem(models.Model):
    # ... other fields ...
    is_active = models.BooleanField(default=True)
    deleted_at = models.DateTimeField(null=True, blank=True)

# "Delete" = set is_active=False, deleted_at=now()
# Query active items: MenuItem.objects.filter(is_active=True)

Why? Hard deletes lose history. If a restaurant "deletes" Pho, you still want to show it in old orders. Soft deletes preserve referential integrity.

Audit Fields

Add created_at and updated_at to every model:

class BaseModel(models.Model):
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        abstract = True

# All your models inherit from this:
class Order(BaseModel):
    # auto_now_add: set once on creation
    # auto_now: updated every time .save() is called
    ...

Unique Constraints

Prevent duplicate data at the database level:

class OrderItem(models.Model):
    order = models.ForeignKey(Order, on_delete=models.CASCADE)
    menu_item = models.ForeignKey(MenuItem, on_delete=models.SET_NULL, null=True)

    class Meta:
        unique_together = [('order', 'menu_item')]
        # Prevents: same menu item added twice to same order
        # Instead: increase quantity

Final Checklist

☑️

Before submitting your schema design:

  1. Did you ask clarifying questions? (6 buckets)
  2. Did you identify access patterns? (screen-by-screen)
  3. Does every entity pass the "entity test"? (attributes + referenced + queryable)
  4. Is every FK on the "many" side?
  5. Did you snapshot data that can change? (price, names on orders)
  6. Did you choose the right on_delete for each FK?
  7. Did you add db_index for filtered/sorted fields?
  8. Does your schema satisfy 1NF, 2NF, 3NF? (with intentional exceptions noted)
  9. Did you add audit fields? (created_at, updated_at)
  10. Did you add __str__ to all models?
  11. Did you consider soft deletes for entities that might be "deleted" but still referenced?

Interview-ready Summary

"My schema design process is: first, I clarify requirements by asking about users, actions, constraints, business rules, edge cases, and scale. Then I identify access patterns by thinking screen-by-screen. From there, I extract entities using the 'look for nouns' technique and validate each one. I map relationships with the rule that FK always goes on the many side. I define attributes, being careful to snapshot any data that can change after an action (like prices on orders). Finally, I verify normalization up to 3NF, add indexes for my access patterns, and apply best practices like audit fields and soft deletes."
